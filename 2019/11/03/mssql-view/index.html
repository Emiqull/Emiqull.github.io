概述

  • 视图是一个虚拟的表，是一个表中的数据经过某种筛选后的显示方式
  • 视图由一个预定义的SELECT查询语句构成
  • 通过视图，客户端不再需要知道底层表的表结构及其之间的关系
  • 视图提供了一个统一访问数据的接口

优点

  • 视图隐藏了底层的表结构，简化了数据访问操作
  • 因为隐藏了底层的表结构，所以加强了安全性，用户只能看到视图提供的数据
  • 使用视图，方便了权限管理，让用户对视图有权限而不是对底层表有权限进一步加强了安全性
  • 视图提供了一个用户访问的接口，当底层表改变后，改变视图的语句来进行适应，使已经建立在这个视图上客户端程序不受影响

分类

普通视图

定义
普通视图由一个Select语句所定义，视图仅仅包含其定义和被引用表的metadata.并不实际存储数据

创建格式
CREATE VIEW view_name AS {select_statement};

创建限制

  • 在View中，除非有TOP关键字，否则不能用Order By子句(若执意要用Order by,可以使用使用Top 100 percent…)
  • View在每个Schema中命名必须独一无二
  • View嵌套不能超过32层（其实实际工作中用不到)
  • Compute,compute by,INTO关键字不允许出现在View中
  • View不能建立在临时表上
  • View不能对全文索引进行查询

索引视图

定义
在普通的视图的基础上，为视图建立唯一聚集索引

创建限制

  • 索引视图涉及的基本表必须ANSI_NULLS设置为ON
  • 索引视图本身必须设置ANSI_NULLS和QUOTED_INDETIFIER为ON
  • 索引视图只能引用基本表
  • SCHEMABINDING必须设置
  • 定义索引视图时必须使用Schema.ViewName这样的全名
  • 索引视图中不能有子查询
  • avg,max,min,stdev,stdevp,var,varp这些聚合函数不能用

创建实例

  • 首先：创建一个基础查询语句

  • 其次：根据该查询语句创建视图

  • 最后：在视图上建立聚集索引

注意
通过普通视图和索引视图的执行计划可以非常清晰的看到索引视图的高性能，但需要注意的是索引视图仅SQL SERVER企业版和开发版才支持

分割视图(分布式分割视图)

定义

  • 整个视图所返回的数据记录，由几个平行表(相同的表结构，表记录不同)所连接的数据集
  • 分割视图总体上可分为两种：本地分割视图、分布式分割视图(本地分割视图仅仅是为了和SQL Server 2005之前的版本的一种向后兼容)

创建限制

  • 分布式分割视图所涉及的表之间的主键不能重复
  • 分布式分割视图中面对的查询中包含聚合函数，尤其是聚合函数中包含DISTINCT，会大大降低性能

创建实例

  • 准备环境：在微软示例数据库AdventureWorks，通过ContactID从前100行和100-200行的数据分别存入两个表，Employee100和Employee200

  • 建立分布式分割视图

注意
通过查看查询视图查找ID是105的执行计划，可以看到查询计划仅查找了Employee200这张表

查询视图定义

SELECT * FROM sys.sql_modules WHERE object_id=OBJECT_ID(‘视图名称’);

刷新视图

前提摘要
普通视图仅仅存储的是select语句和所引用表的metadata，当底层表数据或表结果改变时，有时候视图中表的metadata并没有及时同步，因此需要通过刷新视图手动同步

语法格式
EXEC sp_refreshview 视图名称; –建议在每次修改表结构后，总是刷新相关视图

实践技巧

  • 视图中的SELECT语句性能一定要调到最优；
  • 视图最好不要嵌套，如果非要嵌套，最多嵌套一层
  • 能用存储过程和自定义函数替代视图的，尽量不要使用View视图，存储过程会缓存执行计划，性能更优，限制更少
  • 在分割(分布式分割)视图上，不要使用聚合函数，尤其是聚合函数还包含了Distinct
  • 在建立视图时，如果Where子句能加在视图内，不要加在视图外（因为调用视图会返回所有行，然后再筛选，性能杀手)
  • 视图里尽量避免使用*代表所有列：*不仅对查询性能有影响，而且也不便于视图中关联表结构的更新
  • 无论视图所涉及的表结构有没有修改，在每次对视图中SQL语句修改后，刷新一下视图总是好的
  • 查询视图时，若对视图进行WHERE条件的筛选，默认会在查询时对条件字段建立索引，如果没有的话，大数据量时会很慢
  • 视图中使用类似于TOP 1000且视图外使用WHERE条件筛选，当查询视图时会按照TOP关键字进行数据库系统内部排序，再按照WHERE条件过滤，如果WHERE筛选的数据不在这1000个记录中，就会查询不到数据
